# ⚔️ Fin Adversary Architect: Prism pkg 13 — Round 1 (live-provider MCP foundation)

**VERDICT: Foundation SOUND on isolation (all guardrails held) / NOT YET Round-2-ready — fix ARC-01, ARC-02, ARC-03 before the connector lands or the seam forces rework and a silent precedence skew.**

## Threat Model Summary
- **Topology attacked:** operator → `tools/ProviderDiscovery` CLI → `McpToolSessionFactory` → SDK `McpClient` → provider MCP (Morningstar/Moody's) over Streamable-HTTP + OAuth 2.1. The **running API** does not participate — `AddProviderMcp()` registers a factory nothing resolves.
- **Highest-risk assumption:** "`IMcpToolSession` is the right seam and Round 2 drops in cleanly." It is the right *shape* for parsing, but it ships **no session lifecycle, no fail-soft, and no provider-identity map** — the three things the Round-2 singleton connector actually needs.
- **What genuinely held:** no wiring into the hot path, synthetic stays default, single source of truth (CLI reuses API code), token file git-ignored, no committed secret. Verified below.

## Findings

### [ARC-01] Session lifecycle/pooling is absent from the seam the connector must consume — Severity: High
- **Target:** [backend/FinancialServices.Api/Connectors/Mcp/McpToolSessionFactory.cs](../../../backend/FinancialServices.Api/Connectors/Mcp/McpToolSessionFactory.cs#L30-L50); [IMcpToolSession.cs](../../../backend/FinancialServices.Api/Connectors/Mcp/IMcpToolSession.cs#L24-L34)
- **Attack:** `McpToolSessionFactory.ConnectAsync` returns a bare `IMcpToolSession : IAsyncDisposable` wrapping exactly one `McpClient` (one socket + one `initialize`/`initialized` handshake + one OAuth token acquisition). Round 2's connector is registered `AddSingleton<IProviderRatingsSource>` (see [ServiceCollectionExtensions.cs](../../../backend/FinancialServices.Api/Infrastructure/ServiceCollectionExtensions.cs#L61-L63)). That singleton has two bad choices and no third: **(a)** open a fresh session per `GetRatingAsOfAsync` → every issuer in a parallel sweep pays a full connect+refresh round-trip and a provider under load sees an N-concurrent `initialize` storm; **(b)** cache one long-lived session as a field → now it owns `McpClient`/`HttpClient` lifetime, token-refresh, and **reconnect-on-drop**, none of which `IMcpToolSession` exposes (no health, no keepalive, no get-or-reconnect). There is no pool, no `ISessionProvider`, no "borrow" seam.
- **Impact:** correctness + performance + availability. The connector must invent lifecycle management that belongs in the foundation, and a dropped MCP connection mid-sweep has no defined recovery. This is the central Round-2 rework.
- **Hardening:** add a lifecycle-owning seam now — e.g. `IProviderMcpSessionProvider.GetSessionAsync(ProviderMcpKey, ct)` that owns a single reconnecting session per provider (lazy connect, reconnect on transport fault, dispose on shutdown), so the connector consumes "give me a live session" and never manages `McpClient`. Keep `IMcpToolSession` as the call surface.
- **Best-practice reference:** Well-Architected **Performance Efficiency** (connection reuse) + **Reliability** (defined reconnect). Don't hand a singleton a disposable with no lifecycle.

### [ARC-02] The one resilience rule that matters (§11 fail-soft) is encoded nowhere; no call timeout — Severity: High
- **Target:** [McpToolSessionFactory.cs](../../../backend/FinancialServices.Api/Connectors/Mcp/McpToolSessionFactory.cs#L40-L49); [IMcpToolSession.cs](../../../backend/FinancialServices.Api/Connectors/Mcp/IMcpToolSession.cs#L26-L33)
- **Attack:** planning §11 (non-negotiable) says "live fault = fail-soft to synthetic / MISSING_COVERAGE, never throws on the hot path." But every method on this seam **throws** — `ConnectAsync`/`ListToolsAsync`/`CallToolAsync` surface SDK exceptions, `ReloginRequiredException`, and `ConfigurationException` straight up. And the SDK owns the `HttpClient` in `ConnectAsync` with **no timeout set** — contrast [ServiceCollectionExtensions.cs](../../../backend/FinancialServices.Api/Infrastructure/ServiceCollectionExtensions.cs#L44-L57) where `EdgarClient`/`FredClient` both pin `Timeout = 15s`. A hung provider therefore stalls until the SDK's default `InitializationTimeout` / socket timeout, inside a reconciliation sweep.
- **Impact:** availability. The connector must wrap every call in `try/catch → null` **and** impose its own timeout, or a single slow/broken provider hangs or faults the whole sweep — exactly the §11 failure the foundation was meant to make safe. The resilience contract lives only in prose.
- **Hardening:** give the seam a fail-soft entry point (e.g. `Task<McpToolCallResult?> TryCallToolAsync(...)` that maps transport/SDK faults to `null` and rethrows only `OperationCanceledException`), and set an explicit `HttpClientTransportOptions`/`HttpClient` timeout consistent with the 15s connector convention. Encode "never throw on hot path" in the type, not the comment.
- **Best-practice reference:** Well-Architected **Reliability** (bulkhead/timeout), and repo P7 (cancellation must still propagate).

### [ARC-03] Provider identity is fragmented three ways — a direct trap for the Round-2 ARC-01 precedence guard — Severity: High
- **Target:** [ProviderMcpEndpointGuard.cs](../../../backend/FinancialServices.Api/Connectors/Mcp/ProviderMcpEndpointGuard.cs#L7-L13) (`ProviderMcpKey`); the domain `Provider` enum (`Moodys`/`MorningstarDbrs`/`Msci`); [DiscoveryReportWriter.cs](../../../tools/ProviderDiscovery/DiscoveryReportWriter.cs#L100)
- **Attack:** there are now **three spellings** of the same provider: `ProviderMcpKey.Morningstar` (MCP layer), `Provider.MorningstarDbrs` (domain/`IProviderRatingsSource.Provider`), and config `Providers.Morningstar` / `ProviderApis.MorningstarApi`. `ProviderMcpKey.ToString()` = `"Morningstar"` is what becomes the token-file name, `ReloginRequiredException.Provider`, and the `SourceRef` provenance template (`live:morningstar:<id>`). Round 2's ARC-01 fix is **read-time precedence keyed on the `source`/provider string**; a live doc stamped `"Morningstar"` (or `"morningstar"`) that must out-rank a synthetic `MorningstarDbrs` row will **silently fail to match**, and you get the exact dual-source `Moodys-vs-Moodys` / duplicate-provider skew ARC-01 exists to prevent (see plan-review ARC-01). No `ProviderMcpKey ↔ Provider` adapter exists; the mapping is unwritten and the names don't even agree.
- **Impact:** correctness (silent) + P1/P2. The precedence guard's key is being minted here, in the wrong vocabulary, with no single mapping.
- **Hardening:** add one authoritative map `ProviderMcpKey → Provider` (and a canonical lowercase `source` token) in a single helper now; make the `SourceRef` template and any future `source` field derive from `Provider`, not from `ProviderMcpKey.ToString()`. Unit-test the round-trip.
- **Best-practice reference:** eliminate hidden coupling / dual-source-of-truth (repo P8); this is the seam where the ARC-01 key is born.

### [ARC-04] The Phase-0 CLI cannot pick up credentials as documented — no `.env` loader, no wrapper script — Severity: High (fails safe)
- **Target:** [tools/ProviderDiscovery/Program.cs](../../../tools/ProviderDiscovery/Program.cs#L14-L38); [ProviderDiscovery.csproj](../../../tools/ProviderDiscovery/ProviderDiscovery.csproj) (no `appsettings.json`, no content copy)
- **Attack:** config is read via `builder.Configuration.GetSection("Prism").Get<PrismOptions>()` on a default `Host.CreateApplicationBuilder()`, which reads env vars / appsettings / command line — **not `.env` files**. The header comment says "Reads `Prism__Providers__…` from the environment (load .env first, e.g. via `run-provider-discovery.bat`)", but **that bat does not exist** (only `run-backend.bat` is present) and there is no `appsettings.json` in the project. So out of the box `prism.Providers.Morningstar` is all-defaults (`Enabled=false`), and the tool that exists to clear the go/no-go gate immediately logs "not enabled" unless the operator has *manually* exported `Prism__Providers__*` into the process environment.
- **Impact:** operational excellence. The one runnable Round-1 deliverable is not runnable via its own documented path. It fails **loud and safe** (P1 preserved), so this is readiness, not a live risk — but the happy path is broken.
- **Hardening:** add the missing `run-provider-discovery.bat` that sources `.env`, **or** load `.env` in `Program.cs` (DotNetEnv), **or** ship an `appsettings` + document `--` env passing. Then add a smoke test that the enabled+ClientId gate is reached with vars set.
- **Best-practice reference:** Well-Architected **Operational Excellence** — a tool's documented invocation must work.

### [ARC-05] CLI ProjectReferences the entire API — drags Cosmos/Search/Agents/AGUI for an MCP-only tool — Severity: Medium
- **Target:** [ProviderDiscovery.csproj](../../../tools/ProviderDiscovery/ProviderDiscovery.csproj#L22-L33); DI registration [ServiceCollectionExtensions.cs](../../../backend/FinancialServices.Api/Infrastructure/ServiceCollectionExtensions.cs#L64-L77)
- **Attack:** the CLI needs only `Connectors/Mcp/*` + `PrismOptions` + two exceptions, but `<ProjectReference Include="...FinancialServices.Api.csproj" />` transitively compiles and loads Cosmos 3.46, AI Search, Agent Framework, and the **AGUI prerelease** hosting package. The `<AzureCosmosDisableNewtonsoftJsonCheck>true</AzureCosmosDisableNewtonsoftJsonCheck>` in the csproj is the *symptom*: a tool with no Cosmos need must suppress a Cosmos build guard purely because it drags the web app. Separately, `AddProviderMcp()` registers `McpToolSessionFactory` in the API DI container but **nothing resolves it**, and the CLI builds it with `new McpToolSessionFactory(...)` — so the registration currently serves zero consumers (premature/dead).
- **Impact:** cost (build time, binary surface, prerelease-dep blast radius reaching a CLI) + a misleading DI registration implying the API consumes MCP when it doesn't.
- **Hardening:** extract `Connectors/Mcp/*` into a small `FinancialServices.Connectors` (or `.Mcp`) library referenced by both API and CLI; drop the Cosmos check suppression; either have the CLI resolve the factory from a minimal DI container or defer `AddProviderMcp()` until a real API consumer exists (Round 2).
- **Best-practice reference:** dependency minimization / single-purpose module boundaries.

### [ARC-06] `ReloginRequiredException` advertises a client code the sole mapping layer ignores — Severity: Medium
- **Target:** [ReloginRequiredException.cs](../../../backend/FinancialServices.Api/Infrastructure/Errors/ReloginRequiredException.cs#L13-L26); [PrismExceptionHandler.cs](../../../backend/FinancialServices.Api/Infrastructure/Http/PrismExceptionHandler.cs#L47-L64)
- **Attack:** the exception publishes `ErrorCode = "RELOGIN_REQUIRED"` and a `Code` property, implying an HTTP contract. But `PrismExceptionHandler.Map` has no arm for it — it falls through to `_ => 500 "INTERNAL_ERROR / An unexpected error occurred."` So if Round 2 throws it on any API path, the operator-actionable "re-login" signal becomes an opaque 500. Worse, it's thrown from *inside* the SDK's redirect delegate ([ProviderOAuth.cs](../../../backend/FinancialServices.Api/Connectors/Mcp/ProviderOAuth.cs#L58-L64)); if `McpClient.CreateAsync` wraps delegate faults, the CLI's top-level `catch (ReloginRequiredException)` ([Program.cs](../../../tools/ProviderDiscovery/Program.cs#L113-L117)) also misses it and it lands in the generic catch. Exception-type control flow across a third-party async boundary is unverified.
- **Impact:** correctness of error contract; a promised stable code that doesn't surface. Inert today (no API path throws it), so Medium.
- **Hardening:** add a `ReloginRequiredException` arm to `PrismExceptionHandler.Map` (e.g. 409/401 + `RELOGIN_REQUIRED` + `provider` detail) when Round 2 wires it; in the CLI, also catch the SDK's wrapper and unwrap `InnerException is ReloginRequiredException`. Add a test asserting the mapping.
- **Best-practice reference:** error taxonomy consistency (arch-03) — a type must not advertise a contract the boundary drops.

### [ARC-07] "Key Vault ETag seam is real / nothing else changes" over-claims — the rotation race stays open — Severity: Medium
- **Target:** [FileTokenCache.cs](../../../backend/FinancialServices.Api/Connectors/Mcp/FileTokenCache.cs#L10-L16) (doc comment)
- **Attack:** the class doc says a production swap "swaps in a Key-Vault-backed implementation (ETag-guarded write-back for the rotation race); nothing else changes because the SDK depends only on the interface." But the SDK's `ITokenCache` surface is `GetTokensAsync(ct)` / `StoreTokensAsync(tokens, ct)` — **no ETag, no version, no CAS in or conflict out.** A Key-Vault impl can do *internal* conditional writes, but the SDK gives it no hook to react to a lost race, and two replicas rotating the same refresh token still need **single-flight refresh coordination** the interface cannot express. So "just swap `ITokenCache`" does **not** close the multi-replica race — it only relocates storage.
- **Impact:** a future multi-replica deploy inherits a silent token-clobber → revoked-refresh-token auth failures, under the false comfort that the seam already solved it. Correctly deferred (Round 1 is local-dev-only, decision 8a) — the issue is the over-promise, not the file cache.
- **Hardening:** reword the comment to "storage swap seam; the multi-replica rotation race requires single-flight refresh coordination that is out of scope for local dev and must be designed before any scaled deploy." Track the race as an explicit Round-2+/deploy blocker.
- **Best-practice reference:** Well-Architected **Reliability** — don't mark an unsolved concurrency hazard as solved.

### [ARC-08] Dual config surface for the same providers (`ProviderApis` vs `Providers`) — Severity: Medium
- **Target:** [PrismOptions.cs](../../../backend/FinancialServices.Api/Infrastructure/PrismOptions.cs#L44-L60); still live-read at [MoodysRatingsClient.cs](../../../backend/FinancialServices.Api/Connectors/MoodysRatingsClient.cs#L26-L29)
- **Attack:** `Prism:ProviderApis:{MoodysApi,MorningstarApi}:{BaseUrl,ApiKey}` (old REST placeholders) coexists with the new `Prism:Providers:{Morningstar,Moodys}:{...}` MCP tree. The runtime connectors still read the **old** `ProviderApis` (for their `configured=` log lines), while the new `Providers` tree is consumed only by the CLI. Two config trees describe the same two providers with inconsistent names (`MoodysApi` vs `Moodys`, `MorningstarApi` vs `Morningstar`).
- **Impact:** drift/confusion; Round 2 must migrate the connectors from `ProviderApis` to `Providers`, after which `ProviderApis` is fully dead. A reviewer/operator can't tell which knob is live.
- **Hardening:** mark `ProviderApis` `[Obsolete]`/comment as superseded in code (the `.env.example` already comments it out), and plan its deletion when the connectors switch. Don't let two live trees describe one provider.
- **Best-practice reference:** single source of truth for configuration.

### [ARC-09] `ProvidersOptions` shape + CLI resolution don't scale past two providers — Severity: Medium
- **Target:** [PrismOptions.cs](../../../backend/FinancialServices.Api/Infrastructure/PrismOptions.cs#L90-L98); resolution [Program.cs](../../../tools/ProviderDiscovery/Program.cs#L46); allowlist [ProviderMcpEndpointGuard.cs](../../../backend/FinancialServices.Api/Connectors/Mcp/ProviderMcpEndpointGuard.cs#L36-L41)
- **Attack:** `ProvidersOptions` uses named properties (`Morningstar`, `Moodys`), and the CLI resolves options with a **two-value ternary**: `cli.Provider == ProviderMcpKey.Morningstar ? prism.Providers.Morningstar : prism.Providers.Moodys`. Add the LSEG provider the research already names, and it silently maps to **Moody's** config. Provider identity now lives in **three parallel registries that must stay hand-synced**: the `ProviderMcpKey` enum, the `Allowed` host dict, and the `Providers` named props.
- **Impact:** maintainability + a silent mis-binding on the next provider.
- **Hardening:** key the config by provider (`IReadOnlyDictionary<ProviderMcpKey, ProviderMcpOptions>` or a lookup), and resolve options + hosts from one keyed source so adding a provider touches one place.
- **Best-practice reference:** open/closed — extending providers shouldn't require editing a ternary and three tables.

### [ARC-10] The offline test seam was built and left unexercised — the Round-2 parse path has never run — Severity: Medium
- **Target:** [McpToolSessionFactory.cs](../../../backend/FinancialServices.Api/Connectors/Mcp/McpToolSessionFactory.cs#L54-L72) (`ConnectWithHttpClientAsync`, zero callers); [McpToolSession.cs](../../../backend/FinancialServices.Api/Connectors/Mcp/McpToolSession.cs#L13-L49); [LoopbackAuthorizationHandler.cs](../../../backend/FinancialServices.Api/Connectors/Mcp/LoopbackAuthorizationHandler.cs#L83-L115) (`TryExtractCode`)
- **Attack:** the test project has exactly two files — `ProviderMcpEndpointGuardTests` and `TokenFreshnessTests`. `ConnectWithHttpClientAsync` was purpose-built ("fake handler serves the JSON-RPC, no live network") to drive a real `McpClient` through the parse path — and it has **no caller**. So `McpToolSession`'s SDK→Prism mapping (`TextContentBlock` filtering, `StructuredContent.GetRawText()`, tool projection) — the exact code Round 2's mapper builds on — has never executed. `TryExtractCode` (the **CSRF state-match**, documented "unit-testable, no I/O") is untested. `FileTokenCache` (atomic write / corrupt-JSON degrade / round-trip), `ProviderOAuth.BuildOptions` (fail-loud on missing ClientId, scope minimization, headless throw), and the whole CLI (`CliArgs`, `RepoLayout`, `DiscoveryReportWriter`) are untested; there is **no `ProviderDiscovery.Tests` project** (slnx lists only ProviderDiscovery).
- **Impact:** Round-2 readiness. The seam's confidence is limited to SSRF + freshness math; the parse/mapping/CSRF/token-persistence behaviour is unverified, and prior packages held ~188 tests — this is a discipline regression on the just-written code.
- **Hardening:** add the fake-`HttpMessageHandler` MCP test (JSON + `text/event-stream`) through `ConnectWithHttpClientAsync`, a `TryExtractCode` table (state-match/mismatch/error/missing-code), `FileTokenCache` round-trip + corrupt-degrade, `ProviderOAuth.BuildOptions` fail-loud, and a `ProviderDiscovery.Tests` project for `CliArgs`.
- **Best-practice reference:** if you build a test seam, exercise it; test what Round 2 depends on (arch-11).

### [ARC-11] `TokenFreshness` is tested but never consumed — Severity: Low
- **Target:** [TokenFreshness.cs](../../../backend/FinancialServices.Api/Connectors/Mcp/TokenFreshness.cs); usage grep = tests only
- **Attack:** `NeedsRefresh`/`SecondsUntilExpiry` are called only from `TokenFreshnessTests`. The SDK owns the real silent refresh; no CLI or runtime path calls this helper. It's dead (if harmless, and at least tested) weight added to justify a skew test.
- **Impact:** minor YAGNI / misleading "this is the offline half of the broker" when nothing calls it.
- **Hardening:** wire it into a pre-flight log ("token has Ns of life") in the CLI, or drop it and keep the note that the SDK owns skew.

### [ARC-12] Loopback listener leaks an orphaned task on timeout; no port-in-use pre-flight — Severity: Low
- **Target:** [LoopbackAuthorizationHandler.cs](../../../backend/FinancialServices.Api/Connectors/Mcp/LoopbackAuthorizationHandler.cs#L44-L60)
- **Attack:** on the timeout branch, `listener.GetContextAsync()` is abandoned; disposing the `using` listener aborts it, faulting the orphaned task with no `await` → a possible `UnobservedTaskException`. And the fixed redirect port (`8765`, required to match the registered OAuth redirect) has no pre-flight check — a leaked prior listener or another process on 8765 surfaces as an opaque `HttpListenerException` through the CLI's generic catch, with no actionable message.
- **Impact:** local-dev ergonomics; a confusing failure on a re-run after an interrupted login.
- **Hardening:** observe/swallow the orphaned task (or cancel via the linked CTS and await it), and pre-flight the port with a clear "port 8765 in use — close the previous login" message.

### [ARC-13] Discovery note renders provider-supplied text verbatim into markdown — Severity: Low
- **Target:** [DiscoveryReportWriter.cs](../../../tools/ProviderDiscovery/DiscoveryReportWriter.cs#L48-L64)
- **Attack:** `tool.Description` and `tool.JsonSchema` come straight from the remote MCP server and are written raw into the findings `.md` (description as free text, schema inside a fence). A hostile/compromised server could inject markdown or break the code fence in the local note.
- **Impact:** very low — a local-dev artifact a human reads — but the note is presented as "the server's real advertised contract" while rendering untrusted content verbatim.
- **Hardening:** fence-escape the schema and treat description as literal (or note it's untrusted). Cosmetic; list for completeness.

## What's solid (kept honest — brief)
- **Isolation is genuinely clean** — the hard part is right. MCP types appear only in `Connectors/Mcp/*`, the test project, and the CLI (grep-verified). The reconciliation read path (`SearchCorpus`, `ReconciliationService`, `DossierAssembler`) and the `Moodys`/`MorningstarDbrs` clients reference **zero** MCP types. `AddProviderMcp()` registers a factory **nothing resolves**, so the seam is maximally inert in the running API.
- **SSRF guard is thorough and tested** — https-only, exact-host allowlist, IP/loopback/link-local (incl. `169.254.169.254` metadata) rejection, suffix-spoof (`mcp.morningstar.com.evil.com`) rejection, the discovered-AS-server bound to the registrable domain via `AuthServerSelector`, and the deliberate single loopback-http exception for the OAuth redirect. Mirrors the `EdgarClient` pattern faithfully.
- **Fail-loud on absent creds (P1)** — CLI checks `Enabled` then `ClientId` and returns 1 with an actionable message; `ProviderOAuth.BuildOptions` throws `ConfigurationException` naming the setting; the guard names the setting on every rejection.
- **Single source of truth (P8)** — the CLI reuses the API's guard/OAuth/MCP wrapper via ProjectReference; no forked auth/MCP logic.
- **Protocol version not hand-pinned** (STK-04), scope minimized to `offline_access` (SEC-07), blank secret → `null` (public client), atomic temp-file token write, corrupt cache degrades to "re-login" without leaking contents (P6), logs are ids/counts only, `ct` plumbed through every async call (P7).
- **`.env.example`** carries the `Providers` keys with **blank** `ClientId`/`ClientSecret` (no committed secret); token cache is git-ignored (`.prism/`, `**/.prism/`).

## Guardrail confirmation (explicit)
- **No-wiring guardrail — HELD.** Verified by grep: nothing on the reconciliation/`SearchCorpus`/`DossierAssembler`/`Moodys`/`MorningstarDbrs` path touches any MCP type; the API never resolves `McpToolSessionFactory`; the only consumer is the standalone CLI. The running app's behaviour is unchanged.
- **Synthetic-default guardrail — HELD.** `ProviderMcpOptions.Enabled` defaults `false`; no boot gate; `ProviderApis` clients still return `null → MISSING_COVERAGE`; with all flags off the app behaves exactly as before (no live call on any existing path).
- **Reuse-not-duplicate guardrail (P8) — HELD.** CLI reuses API code; token seam is a plain `ITokenCache` (Key-Vault swap point), though ARC-07 notes the swap doesn't by itself close the multi-replica race the comment claims.

## Top 3 Must-Fix (before the Round-2 connector is built on this seam)
1. **ARC-01** — add a lifecycle-owning session seam (per-provider reconnecting session provider); don't hand the singleton connector a bare disposable `McpClient`.
2. **ARC-03** — add one authoritative `ProviderMcpKey → Provider` / canonical `source` map and derive `SourceRef` from it, so the Round-2 ARC-01 read-time precedence guard can't be defeated by a `"Morningstar"` vs `"MorningstarDbrs"` string mismatch.
3. **ARC-02** — encode §11 fail-soft in the type (a `Try*`/nullable call surface) and set an explicit MCP call timeout consistent with the 15s connector convention, so a slow/broken provider can't stall or fault a sweep.
   - *(Close behind: **ARC-10** — exercise the offline `ConnectWithHttpClientAsync` parse path and `TryExtractCode` before Round 2 maps on top of them.)*
